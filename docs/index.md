## Custom Project

### Dataset

The dataset used by the Kafka producer is a comma-separated values file named sales.csv located in the data/ folder. It contains historical commercial transaction logs generated across retail storefront environments. Each record within the dataset includes the following fields: order_id, datetime, region_id, currency_code, product_id, unit_price, quantity, is_online, customer_id, and payment_method. The original sales dataset was kept completely intact and unmodified to serve as a clean baseline for the monitoring pipeline.

### Kafka Messages

The producer reads lines from sales.csv and serializes each row into a raw JSON data payload sent through Kafka. The messages are sent over the default Kafka topic named product-sales-case. The message partition key uses the region_id field to guarantee that all events originating from the same geographic area land on the same partition, preserving strict chronological order. No message fields were altered from the core dataset structural payload.

### Consumer Processing

The consumer receives streaming transaction dictionaries directly from the active Kafka broker partitions. It processes all available messages starting from the beginning of the topic sequence. For every message consumed, it logs specific dictionary payloads, contract schema validations, database insertion confirmations, and active running summaries such as current transaction averages and maximum observed spikes. Validated records are appended to a text file at data/output/consumed_sales.csv. The consumer script filters and transforms incoming field elements to calculate subtotals and tax thresholds, routing the outputs to an isolated custom database asset called data/output/loyalty.duckdb.

### Experiments

* Phase 4 Modification: The streaming live visualization format was changed inside src/streaming/visualizations/live_visualizations_hajiyev.py from a continuous connected line plot to a discrete teal bar chart configuration using axis.bar().
* Phase 5 Application: A data transformation multiplier was implemented inside update_live_chart to translate the raw transaction total into loyalty points (multiplying message[total] by 10). This dynamically converts retail totals into loyalty metric counts without modifying the source dataset files.

### Results

When executing src/streaming/kafka_consumer_hajiyev_custom.py, the consumer established a stable network link to the local bootstrap cluster and initialized an empty tracking dashboard window. Running the producer sent batch rows to the topic cluster queue. As each message landed, the user terminal printed streaming payload logs, and the dashboard window successfully rendered a dynamic, real-time bar graph displaying loyalty points earned, saving the final execution state to data/output/loyalty_chart_hajiyev.png.

### Interpretation

The Kafka streaming workflow demonstrates how micro-batch and real-time messaging frameworks handle data in motion efficiently. 

* What changed: The project pivoted from a generic storefront tracking tool into an isolated premium loyalty and fraud monitoring asset without needing to restructure or alter the core sales dataset columns.
* What was learned: Processing streaming data requires precise coordination between producers and consumer threads. Applying mathematical transformations inside the memory loop allows an application to calculate new operational insights instantly without slowing down transaction processing velocity.
* Organizational value: This architecture allows a business to build non-invasive security logs and user reward updates. Rather than overloading a central production database with recurring analytical queries, a company can filter, convert, and store metrics stream-side.
* Business intelligence gained: This pipeline enables an organization to track customer tier milestones instantly as events occur. By viewing transaction values transformed into loyalty spikes, fraud analysts can catch abnormal purchasing behaviors immediately.
